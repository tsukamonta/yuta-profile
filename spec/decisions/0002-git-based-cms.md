# 0002: Gitベースの静的CMS

- 状態: 保留
- 検討開始日: 2026-06-13

## 保留理由

初期スコープはプロフィールサイトの構築とし、ブログシステムへの拡張は完成後に判断する。
プロフィール各セクションの更新は、CMSではなく、構造化データとMarkdown等のリポジトリ内ファイルで行う。

この記録は将来ブログや編集UIが必要になった場合の比較材料として残す。

## 背景

- CMSは将来の拡張候補であり、初期スコープには含めない
- コンテンツ保存のための常時稼働DBとサーバー費用を避けたい
- 編集者はYuta本人のみを想定する
- プロフィールサイトのコンテンツはMarkdown等のファイルとしてGit管理する

## 静的CMSの考え方

GitベースCMSでは、記事をDBのレコードではなく、Gitリポジトリ内のMarkdown、JSON、YAML、画像として保存する。
CMSはコンテンツを配信するサーバーではなく、ファイルを編集してGitへ保存するUIとして機能する。

```text
CMSで編集
  -> GitHubへファイルを保存
  -> プロフィールサイトを再ビルド
  -> Cloudflare Pagesへ公開
```

この方式では、公開サイト用のコンテンツDBは不要になる。
ただし、公開されたCMSをブラウザから使う場合は、GitHubへの書き込み権限を安全に扱うための認証が必要になる。

## 運用候補

### ローカルCMS

- ローカルで管理画面を起動し、手元のMarkdown等を直接編集する
- 保存後は通常のGit操作でpushする
- 公開CMS、OAuth、CMS用サーバー、DBが不要
- 1人運用では最も構成が小さく、固定費を抑えやすい
- CMSを自作する学習範囲は、編集UI、スキーマ、ファイル操作が中心になる

### GitHub連携CMS

- 公開された管理画面からGitHub APIを通じてファイルを更新する
- 場所を選ばずブラウザ編集できる
- OAuth、アクセストークン、権限管理、コールバック処理が必要
- DBは不要でも、認証処理を動かすサーバーレス機能などが必要になる場合がある

## 既存実装の例

### Decap CMS

- Gitワークフロー向けのオープンソースCMS
- コンテンツをコードと同じリポジトリに保存する
- GitHub、GitLab、Bitbucket等のAPIを利用する
- 静的サイトジェネレーターに依存しない

### Keystatic

- Astro、Next.js等へ組み込める
- ローカルモードでは、管理画面からリポジトリ内のファイルを直接更新できる
- 公開環境で管理画面を動かす場合はサーバー側コードが必要

### Pages CMS

- GitHubリポジトリ内の静的サイト向けコンテンツを直接編集する
- コンテンツ用の別DBを持たない
- CMS自体を自作せず、編集体験だけ追加する選択肢になる

## 将来再検討する場合の評価

一人運用かつサーバー費用を最小化するなら、ローカルCMSが最も単純である。
ブラウザ編集が必要になった時点で、GitHub連携と認証を追加する段階的な構成が適している。

CMSを学習目的で自作する場合も、最初はローカルでファイルを編集するアプリとして作り、プロフィールサイトとはリポジトリまたは書き込み先の境界を分ける。

## 再検討時の判断事項

- CMSをWebアプリとして作るか、ローカルアプリとして作るか
- CMSの自作範囲
- GitHubへの保存をCMSが直接行うか、ローカルGit操作に任せるか
- Markdown等のコンテンツスキーマ
