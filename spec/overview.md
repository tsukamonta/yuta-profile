# プロフィールサイト - 仕様書

## プロジェクト概要

Yutaのプロフィール、能力・趣向、成果物を伝えるためのポートフォリオサイト。
トップページだけで主要情報を把握でき、詳細情報が増えた項目は個別ページへ拡張できる構成とする。
フロントエンド開発を実践的に学ぶ題材として構築する。

技術選定の候補、比較、決定理由は [`decisions/`](./decisions/) に記録する。

## 目的

### サイト

- 訪問者がYutaのプロフィール、能力・趣向、成果物を把握できる
- モバイル、タブレット、PCで支障なく閲覧できる
- セマンティクス、アクセシビリティ、SEO、パフォーマンスを考慮する

### 学習

- HTML、CSS、TypeScript / JavaScriptの基礎を実装と結び付けて学ぶ
- コンポーネント設計と宣言的UIを学ぶ
- Reactを将来的に実務で扱える水準まで学ぶ
- Git、ビルド、自動デプロイを含む開発フローを経験する
- Google Analytics 4 と Google Search Console の使い方を学び、計測と改善の流れを理解する

## ページ構成

| ページ | 役割 |
|---|---|
| トップ | 自己紹介、外部リンク、ページ内目次、プロフィール、能力・趣向、成果物、技術概要をまとめて表示する |

- 初期構成はトップページ1ページとし、成果物の個別ページや一覧ページは設けない
- サイトヘッダーはページ遷移用のグローバルナビゲーションとする
- トップページ内のセクション移動は、ヘッダーとは分離したページ内目次で行う
- Heroには自己紹介と、X / Twitterなどの外部リンクを置く
- 外部リンクと役割が重複するため、独立した連絡先セクションは設けない
- プロフィールは「現在地」「これまでの歩み」に分ける
- 「これまでの歩み」は小見出しを置いて時期別に分ける
- 能力・趣向は「IT・しごと」「文化的趣味・生活」に分ける
- 「IT・しごと」にはスキルと資格を置く
- 「文化的趣味・生活」には趣味や日常的な関心を置く
- 成果物はトップページ内に集約し、小見出し単位で複数区分を追加できる構造にする
- Technical Snapshotは主要コンテンツの後に置く

## 構築・運用方針

- Next.jsのApp RouterとTypeScriptを使用する
- Next.jsは静的エクスポート（`output: "export"`）でビルドする
- 静的サイトとしてビルドし、実行時サーバーを必要としない構成を基本とする
- ブラウザへ送るJavaScriptは、操作に必要な範囲に抑える
- サイトの更新者はYuta本人のみとする
- プロフィールサイトにはブラウザ編集機能を設けない
- プロフィール、能力・趣向、資格、成果物、外部リンクなどの内容は、コンポーネントから分離した構造化データとして管理する
- 初期段階のコンテンツは `src/content/site.ts` に集約し、TypeScriptの型で形式を検証する
- ジャンル、公開状態、画面ラベルなどの固定語彙は `src/content/site-constants.ts` に分離し、本文コンテンツとは別に管理する
- 画面に表示する見出しやラベルは `site.ts` または `site-constants.ts` を情報源とし、TSXへ重複して直書きしない
- 外部リンク、成果物リンク、本文中の外部URLリンクは新しいタブで開き、`rel="noreferrer"` を付与する
- ページ内リンクや内部パスへのリンクは同じタブで遷移する
- 本文中のMarkdown形式リンクは `react-markdown` で描画し、許可するHTML要素はリンク用途に限定する
- TSXはHTML構造、要素ID、CSSクラス、描画処理を担当する
- 長文コンテンツが増えて `site.ts` での管理が難しくなった場合に、Markdown等への分離を再検討する
- Google Analytics 4 は `NEXT_PUBLIC_GA_MEASUREMENT_ID` が設定されている場合のみ有効化する
- Google Analytics 4 の導入に必要な外部接続先は `public/_headers` のCSPに明示する
- Google Search Console は `https://yu-ta.jp/` のURL-prefixプロパティとして扱い、`GOOGLE_SITE_VERIFICATION` によるHTML meta tag確認を基本にする
- `blog.yu-ta.jp` はプロフィールサイトとは別資産として扱い、必要になった時点で別プロパティとして追加する
- コンテンツ更新時はサイトを再ビルドして公開する
- ランニングコストは原則無料、または既存契約の範囲に抑える
- Server Actions、ISR、Cookieを使う認証など、実行時サーバーを必要とする機能は初期スコープで使用しない

## 公開・更新

- ソースコードとコンテンツはGitHubで管理する
- 公開先はCloudflare Pagesとする
- GitHubへのpushを契機に、自動でビルドとデプロイを行う
- Pages Functionsは使用せず、静的アセットのみを配信する

```text
ローカルで編集
  -> GitHubへpush
  -> Cloudflare Pagesでビルド
  -> 静的ファイルを公開
```

## 将来の拡張

- コンテンツ量が増えた項目は、トップページで概要を確認できる状態を保ちながら個別ページへ拡張する
- ブログ機能は初期スコープに含めず、プロフィールサイト完成後に必要性を判断する
- CMSは初期スコープに含めない
- ブログやCMSを追加する場合も、静的サイトとしての単純な配信と低コスト運用を維持できる構成を優先する
